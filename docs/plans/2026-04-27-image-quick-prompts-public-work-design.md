# 2026-04-27 图像快捷提示词与公开作品元数据设计

## 背景

当前画图页提供“优化提示词”功能，但用户更需要的是可复用的快捷提示词入口，以及更完整的公开作品详情信息。本次调整目标是移除提示词优化链路，改为围绕提示词复用和作品元数据展示做最小改动。

## 目标

1. 移除前端“优化提示词”功能及入口。
2. 在画图输入区新增“快捷提示词”能力。
3. 提供 5 条公共参考提示词，作为所有用户可见的快捷入口。
4. 支持从作品详情页把提示词加入“我的快捷提示词”。
5. 用户加入快捷提示词时可以自定义显示名称。
6. 用户可以删除自己加入的快捷提示词。
7. 点击快捷提示词后直接覆盖当前输入框内容。
8. 公开作品创建时使用 `gpt-5-3-mini` 生成简短标题并持久化。
9. 公开作品详情中展示图片文件大小。

## 非目标

1. 不做快捷提示词跨账号、跨设备同步。
2. 不做快捷提示词的后端存储。
3. 不支持自定义快捷提示词的二次重命名。
4. 不做完整的快捷提示词管理页或复杂管理弹窗。
5. 不在本次引入更多提示词编辑能力。

## 方案概览

### 1. 快捷提示词分层

快捷提示词分成两类：

- 公共参考提示词：前端常量，所有用户一致，只可使用，不可删除。
- 我的快捷提示词：前端 `localStorage` 持久化，仅当前浏览器可见，可删除。

这样可以保证同一分享链接被不同人访问时，每个人看到的自定义快捷提示词都不同，符合用户预期。

### 2. 公开作品元数据分层

公开作品标题和文件大小属于作品元数据，应在后端公开作品创建时生成并保存，而不是在前端详情页临时计算。

- 标题：作品公开时用 `gpt-5-3-mini` 根据提示词生成并持久化。
- 文件大小：保存公开图片时直接记录字节数并持久化。

这样同一作品在任何时间、任何终端看到的标题和大小都一致，也避免详情页反复调用模型。

## 详细设计

### 一、移除优化提示词能力

前端删除以下内容：

- 画图页中的“优化提示词”按钮
- 优化结果预览弹窗
- 对 `/api/image/prompts/optimize` 的前端调用

后端优化提示词接口建议一并删除，避免残留死能力。

### 二、画图页快捷提示词

#### 展示位置

在画图页输入组件中新增“快捷提示词”区域，和当前输入区保持同一视觉上下文。

区域分为两组：

- 公共参考
- 我的快捷提示词

#### 交互规则

- 点击任一快捷提示词：直接覆盖当前输入框内容。
- 覆盖后保留用户继续手动编辑能力。
- 删除仅对“我的快捷提示词”可用。
- 公共参考提示词不可删除。

#### 公共参考提示词

公共参考提示词由前端常量提供，初始来源为用户指定的 GitHub 页面中的 5 条不同风格 prompt。

要求：

- 风格明显不同
- 作为只读固定数据使用
- 所有人一致

建议覆盖方向：

- 写实摄影
- 插画 / 二次元
- 电影感海报
- 3D / CG
- 产品 / 极简视觉

#### 我的快捷提示词

使用 `localStorage` 保存，建议 key 为：

`chatgpt2api:image_quick_prompts`

数据结构：

```ts
type CustomQuickPrompt = {
  id: string;
  name: string;
  prompt: string;
  createdAt: string;
}
```

规则：

- 最多保存 30 条。
- 按 `prompt` 内容去重，不按 `name` 去重。
- 若相同 prompt 再次加入，不新增，直接更新显示名称。
- 若 prompt 为空，不允许加入。
- 若名称为空，不允许提交。

### 三、从作品详情页加入快捷提示词

在公开作品详情弹层的提示词卡片中新增“加入快捷操作”按钮。

点击后弹出轻量对话框：

- 展示当前 prompt（只读）
- 输入显示名称（必填）
- 确认后保存到 `localStorage`

行为规则：

- 相同 prompt 已存在：更新名称并提示“已更新快捷提示词名称”
- 不存在：新增并提示“已加入快捷提示词”
- 成功后关闭弹窗

### 四、公开作品标题生成

#### 生成时机

在公开作品创建成功、保存图片前后同一条后端链路中生成标题并入库。

#### 模型

使用 `gpt-5-3-mini`。

#### 输入

以用户原始 prompt 为主，必要时可附带 `revised_prompt` 作为辅助上下文。

#### 输出约束

标题应为简短中文名称，建议控制在 8-16 字，避免以下问题：

- 不要引号
- 不要句号
- 不要无信息量词堆砌
- 不要过长

如果模型调用失败或返回空值，回退为：

`未命名作品`

### 五、公开作品文件大小

后端在保存公开图片字节时，直接记录：

- `file_size_bytes = len(image_data)`

前端详情页展示格式：

- 小于 1024 KB：显示 `xxx KB`
- 否则显示 `x.xx MB`

旧数据无该字段时显示 `—`。

## 数据模型变更

### 前端 `PublicWork`

在现有字段基础上新增：

```ts
title?: string;
file_size_bytes?: number;
```

### 后端公开作品持久化字段

在现有公开作品结构基础上新增：

- `title`
- `file_size_bytes`

需要保证以下存储后端都兼容：

- JSON 存储
- Git 存储
- 数据库存储

## 兼容性与风险

1. 旧公开作品没有 `title` 和 `file_size_bytes`，前端必须兼容空值。
2. 公共参考提示词来自外部文档，落地时需确保实际挑选内容风格差异足够明显。
3. 删除优化提示词功能后，所有入口和无用 API 调用必须清理干净，避免悬空能力。
4. 标题生成属于额外模型调用，必须保证失败时不影响公开作品主流程。

## 测试范围

### 前端

1. 点击公共参考提示词可覆盖输入框。
2. 点击我的快捷提示词可覆盖输入框。
3. 作品详情页可新增快捷提示词。
4. 相同 prompt 重复加入时只更新名称。
5. 自定义快捷提示词可删除。
6. 删除优化提示词后页面无残留按钮和弹窗。
7. 作品详情页展示标题、尺寸、创建时间、文件大小。

### 后端

1. 公开作品创建时能写入 `title`。
2. 公开作品创建时能写入 `file_size_bytes`。
3. 标题生成失败时回退为 `未命名作品`，且不影响公开作品保存。
4. `GET /api/public-works` 和 `GET /api/public-works/{work_id}` 返回新增字段。
5. 旧数据缺失新增字段时接口仍可正常返回。
